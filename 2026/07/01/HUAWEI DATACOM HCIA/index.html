Overview

OSI/RM

OSI/RM

  • 应用层 Application Layer

    实现功能：1.满足下层模型无法解决的问题 2.产生不同的数据 3.验证手段、加密手段

  • 表示层 Presentation Layer

    实现功能：负责数据的编译、解码

  • 会话层 Session Layer

    实现功能：可以实现应用级别不同服务进程间的通信

  • 传输层 Transport Layer

    实现功能：1.TCP UDP 按照协议决定是否实现纠错功能 2.区分不同设备【端口】

    无法实现：1.有可能无法实现纠错 2.无法实现应用级别不同服务的区分

  • 网络层 Network Layer

    实现功能：1.从全局的角度上定位设备【IP】 2.路由

    无法实现：1.纠错 2.无法实现一个设备不同应用的区分

  • 数据链路层 Data Link Layer

    实现功能：1.差错检测 2.提供链路层地址信息【MAC】

    无法实现：1.纠错 2.无法从全局的角度分析地址信息

  • 物理层 Physical Layer

    作用：规范传输介质的标准

    传输介质：光纤、双绞线、自由空间

    实现功能：1.实现了设备的连接 2.实现了数据的传输

    无法实现：1.差错检测

总结：OSI/RM的每一层定义十分明确、各司其职，下层永远为上层提供服务

问题：1. 每一层过于明确，导致应用开发相对复杂 2.OSI/RM定义了每一层的职责，但是没有定义每一层的标准、协议

Transport Layer TCP

Overview

传输层：实现“端到端”的服务，即应用到应用的服务

TCP Transmission Control Protocol

  1. TCP是一种面向连接的传输层协议，可提供可靠的传输服务。
  2. TCP只能适用于单播通信。
  3. TCP基于已经建立的连接，实现对数据的控制。
  4. TCP无法满足即时性要求，适合稳定性要求高的流量。
  5. TCP可以感知上层数据的大小，有分片机制，用于相对大的数据。

UDP User Datagram Protocol

  1. UDP是一种面向无连接的传输层协议，传输可靠性没有保证。
  2. UDP可以用于单播、多播和广播。
  3. UDP无法实现对数据的控制，如果想控制，交给应用层。
  4. UDP满足即时性要求，适合时间要求高的流量。
  5. UDP全盘接收上层的数据，没有分片机制，用于相对小的数据。

ack有两个作用：

一是同意与对方建立连接

二是希望对方下次以该数据作为seq，为了接下来数据传输的有序性

数据传输过程

确认机制

ack的值

三次握手时，ack为对方的seq+1

数据传输时，ack为对方的seq+对方所传输的数据长度，或者，对方所传递的数据最大值+1

seq的值

三次握手前两个包为随机值

数据传输时（ACK==1 SYN == 0），seq为对方的ack

TCP很可靠 可靠到我发给你的包 我都知道你收到了

有序机制

TCP会对应用层交付的数据进行分片和排序

MSS Maximum segment size 这个选项在握手的前两个包中协商好

重传机制

进行延迟确认 (Delayed ACK)和丢包时的策略：立即发送重复 ACK

![image-20260530132202998](markdown-img/HUAWEI DATACOM HCIA.assets/image-20260530132202998.png)

超时重传机制

如果发送了数据，一直没有得到对方的确认，在一定时间周期后，重发。【RTTS】

流量控制 window

在数据发送方，有发送缓存区，窗口中存放的是1.发送但未被确认的数据 2.未发送的数据

在数据接收方，有接收缓存区，窗口中存放的是 1.未交付的数据

接收方调整对方的窗口，以此来影响发送方的发送多少

OSPF

概述

从内外部网关协议划分

IGP协议 内部网关协议 AS内部 OSPF/IS-IS

EGP协议 外部网关协议 AS外部 BGP

从协议算法划分

链路状态路由协议 OSPF/IS-IS

特点：传递设备的详细内容，包括拓扑信息、连接状态、开销信息等，每台设备可以根据信息构建拓扑，从而知道网络中路由的详细内容。

距离矢量路由协议 RIP

特点：只知道路由的传递，但是不知道路由的具体始发信息，逐跳收敛。

路径矢量路由协议 BGP

备注

  1. AS：即自治系统，分为公有AS和私有AS
  2. 矢量行为：协议收到一个路由后，如果可以加表（可达），就传递。

OSPF原理介绍

SPF 最短路径优先算法

运行OSPF的设备，会产生一些参数：

  1. LSA 链路状态通告
  2. LSDB 链路状态数据库 // 存放LSA的地点
  3. 邻居/邻接 必须保障设备可以建立关系，才可以进行同步

三大过程：

  1. 建立邻居、邻接关系
  2. 同步LSA，构建LSDB
  3. 根据SPF计算路由

OSPF报文

![image-20260531123522144](markdown-img/HUAWEI DATACOM HCIA.assets/image-20260531123522144.png)

OSPF报文封装在IP报文中，协议号为89。

OSPF报文类型有5种：

  1. Hello 报文 实现邻居关系的建立

    收到对方的hello包后，会将对方的rid添加到active neighbor中

    对方看到这个hello包，进入2-way状态

  2. DD（Database Description）报文

    用于描述LSDB的摘要信息

    DD seq —— DD报文的序列号，只有主设备会创建，从设备不会创建，即序列号由主定。

    主从关系的确定，比较二者的RID，大的为主。

  3. LSR（Link State Request）报文

    LSR用于向对方请求所需的LSA。

  4. LSU（Link State Update）报文

    LSU用于向对方发送其所需要的LSA。

  5. LSACK（Link State Acknowledgment）报文

    LSACK用于向对方发送收到LSA的确认。

需要配置参数：

  1. OSPF的process-id

  2. OSPF的router-id

    可以人为配置

    可以自动产生（系统的rid，这是huawei特有的，给所有用rid的协议准备的）

    第一个up的接口，该接口地址成为rid

    如果多个接口up，优选ip地址大的，形成全局rid

    rid一经确定，除非设备重置或者OSPF协议重置，否则不会改变

  3. OSPF的Area号

    LSA同步的前提条件：处于相同的区域中

    用于控制每个区域的LSDB大小，节约性能。

    骨干区域 area == 0

    非骨干区域 area != 0

  4. OSPF通过接口建立邻居关系

    宣告！

    • 区域下宣告 network 接口信息 + 反掩码信息

    • 接口下宣告

      必须保障全局的ospf协议建立了该进程 以及 创建了该区域

OSPF用到两个组播地址

其中 224.0.0.5 所有人建立邻居关系都会发送该地址

OSPF的七个状态

当前的OSPF协议与其他设备的OSPF协议处于怎样的状态

Down

Init

2-way 处于2-way状态 就是邻居关系

Exstart

Exchange

Loading

Full 处于Full状态，就是邻接关系

down：没有发现任何设备

init 是收到了对方的hello包，但是没有在active neighbor中看到自己

2-way是收到了对方的hello包，并且在active neighbor中看到了自己

exstart 主从关系选举，使用不包含任何LSDB摘要信息的DD报文——“空DD报文”

从exstart状态，解决了两个问题：

一是LSDB同步时，不同步重复的LSA

二是实现了同步的可靠（通过seq实现了确认）

exchange 发送的DD报文中，包含LSA的头部信息，用来描述LSDB的摘要信息。

为什么需要选举

同一个广播域中，路由器的数量越多，建立的邻接关系越多，交互的报文越多，存在问题：

  1. 占用过多的链路带宽
  2. 浪费过多的设备性能

所以我们在2-way之后，exstart之前选举 DR（指定路由器） BDR（备份指定路由器）

DR BDR会和所有的设备建立full邻接关系

DR-OTHER之间只会建立2-way邻居关系

以太网链路，默认是BMA网络类型，BMA网络必须选举DR/BDR（当然了，BDR可以没有）

如何选举DR/BDR？

  1. 接口优先级【包含在hello报文中】 默认1 范围0-255

    如果人为修改为0，不参与选举

  2. 设备的RID【通用头部】

    现网，设备性能高的指定为DR

注意：

一个BMA网络中，接口优先级最高的设备，不一定成为DR

接口从运行OSPF后，就开始选举DR BDR，存在一个选举时间 wait 的概念（40s），40s内需要选出DR/BDR。

VLAN

VLAN解决的问题

随着网络中计算机的数量越来越多，传统的以太网络开始面临冲突严重、广播泛滥以及安全性无法保障等各种问题。

VLAN（Virtual Local Area Network）即虚拟局域网，是将一个物理的局域网在逻辑上划分成多个广播域的技术。通过在交换机上配置VLAN，可以实现在同一个VLAN内的用户可以进行二层互访，而不同VLAN间的用户被二层隔离。这样既能够隔离广播域，又能够提升网络的安全性。

端口/链路类型

![image-20260603215131087](markdown-img/HUAWEI DATACOM HCIA.assets/image-20260603215131087.png)

用户主机和交换机之间的链路为接入链路，交换机与交换机之间的链路为干道链路。

PVID表示端口在缺省情况下所属的VLAN。

缺省情况下，X7系列交换机每个端口的PVID都是1。

端口类型 收到不带Tag的帧 收到带Tag的帧 发送 作用
ACCESS 接收，打上该端口PVID 查看VLAN ID，与PVID相同则接收，不同则丢弃 查看VLAN ID，与PVID相同则去标签发送，不同则丢弃 终端直连
TRUNK 添加端口的PVID，在VLAN ID列表中接收，否则丢弃 在VLAN ID列表中接收，否则丢弃 在VLAN ID列表中，且VLAN ID与PVID相同，去Tag发送；在VLAN ID列表中，且VLAN ID与PVID不同，保持原有Tag发送 交换机互联
BYBRID 同TRUNK 同TRUNK 当VLAN ID是允许通过的，发送，且可以通过命令设置发送时是否携带Tag。

什么时候会收到带Tag的帧呢？

连接虚拟化服务器，存在VSW（虚拟交换机）

ACCESS端口，只能属于一个VLAN，只能为一个VLAN打标签/剥离标签。

如果该端口收到对端设备发送的帧是untagged（不带VLAN标签），交换机将强制加上该端口的PVID。如果该端口收到对端设备发送的帧是tagged（带VLAN标签），交换机会检查该标签内的VLAN ID。当VLAN ID与该端口的PVID相同时，接收该报文。当VLAN ID与该端口的PVID不同时，丢弃该报文。

TRUNK端口，有个VLAN ID列表，只能为一个VLAN剥离标签。

当接收到对端设备发送的不带Tag的数据帧时，会添加该端口的PVID，如果PVID在允许通过的VLAN ID列表中，则接收该报文，否则丢弃该报文。当接收到对端设备发送的带Tag的数据帧时，检查VLAN ID是否在允许通过的VLAN ID列表中。如果VLAN ID在接口允许通过的VLAN ID列表中，则接收该报文。否则丢弃该报文。

端口发送数据帧时，当VLAN ID与端口的PVID相同，且是该端口允许通过的VLAN ID时，去掉Tag，发送该报文。当VLAN ID与端口的PVID不同，且是该端口允许通过的VLAN ID时，保持原有Tag，发送该报文。

BYBRID端口，

Hybrid端口是交换机上既可以连接用户主机，又可以连接其他交换机的端口。Hybrid端口既可以连接接入链路又可以连接干道链路。Hybrid端口允许多个VLAN的帧通过，并可以在出端口方向将某些VLAN帧的Tag剥掉。华为设备默认的端口类型是Hybrid。
